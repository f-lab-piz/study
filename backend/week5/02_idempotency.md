## 멱등성이란?

멱등성은 **같은 요청이 여러 번 들어와도 결과가 안전해야 한다**는 개념이다.

쉽게 말하면:

> 한 번 실행하든 여러 번 실행하든 시스템이 꼬이지 않아야 한다.

---

## 왜 멱등성이 필요한가?

실제 서비스에서는 같은 요청이 한 번만 온다고 가정하면 안 된다.

이유:

- 사용자가 버튼을 두 번 누를 수 있다
- 모바일 네트워크가 불안정할 수 있다
- 클라이언트가 타임아웃 후 재시도할 수 있다
- 서버나 프록시가 재요청할 수 있다

즉, **중복 요청은 흔한 일**이다.

---

## 물류에서 어떤 문제가 생길까?

예를 들어 `POST /picking-tasks` 요청이 두 번 들어왔다고 생각해보자.

그러면 아래 문제가 생길 수 있다.

- 작업지시가 두 개 생성된다
- 같은 재고가 두 번 차감된다
- 운영자는 어떤 작업이 진짜인지 헷갈린다

즉, 멱등성이 없으면 데이터보다 **업무 흐름**이 망가진다.

---

## 예시로 이해하기

### 안전하지 않은 경우

```text
POST /outbound-requests
```

사용자가 저장 버튼을 두 번 눌렀다.

결과:

- 출고 요청이 2개 생성됨

### 더 안전한 방향

같은 요청이라는 것을 시스템이 구분할 수 있어야 한다.

예:

- 같은 `request_id`
- 같은 `idempotency_key`
- 같은 외부 주문 번호

그러면 이미 처리한 요청인지 확인할 수 있다.

---

## 어떤 API에서 특히 중요할까?

멱등성은 모든 API에서 중요하지만, 특히 아래에서 더 중요하다.

- 생성 API
- 결제 API
- 재고 차감 API
- 작업지시 생성 API

물류 예시:

- `POST /outbound-requests`
- `POST /picking-tasks`

---

## 상태 관리와 같이 보면 더 쉽다

상태 관리와 멱등성을 같이 보면 실무 감각이 좋아진다.

예를 들어:

- 이미 `completed` 인 작업을 다시 완료 처리
- 이미 생성된 작업지시를 또 생성
- 이미 실패한 요청을 같은 키로 다시 생성

이런 상황을 막아야 시스템이 안정적이다.

즉:

- 상태 관리는 "지금 어디 있는가"
- 멱등성은 "같은 요청이 다시 와도 괜찮은가"

를 다룬다.

---

## 멱등성을 어떻게 생각하면 좋을까?

이번 주에는 구현 디테일보다는 아래 질문에 익숙해지는 것이 중요하다.

1. 이 API는 중복 호출되면 위험한가?
2. 같은 요청인지 어떻게 구분할 수 있을까?
3. 이미 처리한 요청이면 어떤 응답을 줄까?

---

## 간단한 판단 기준

### 중복 호출이 위험한 API

- 새 데이터 생성
- 재고 차감
- 외부 시스템 전송
- 작업지시 생성

### 중복 호출이 비교적 덜 위험한 API

- 조회 API
- 단순 목록 조회
- 상태 확인

---

## 물류 예시

### 예시 1. 출고 요청 생성

같은 외부 주문 번호로 출고 요청이 두 번 오면:

- 새 요청을 또 만들지 않아야 한다
- 기존 요청을 반환하거나
- 이미 처리된 요청이라고 알려줘야 한다

### 예시 2. 작업지시 생성

같은 출고 요청으로 작업지시를 두 번 만들면:

- 현장 작업이 중복된다
- 재고도 틀어질 수 있다

즉, 작업지시 생성은 멱등성이 꼭 필요한 API다.

---

## 테스트로 어떻게 확인할까?

아래 같은 테스트를 생각해볼 수 있다.

- 같은 요청을 두 번 보내도 결과가 하나만 생기는지
- 같은 키로 다시 요청했을 때 기존 결과를 돌려주는지
- 중복 생성이 막히는지

이렇게 보면 멱등성도 테스트 가능한 요구사항이 된다.

---

## 정리

- 멱등성은 중복 요청을 안전하게 처리하는 개념이다
- 실서비스에서는 중복 요청이 자주 생긴다
- 물류에서는 생성/차감/작업지시 API에서 특히 중요하다
- 상태 관리와 같이 보면 더 이해하기 쉽다
