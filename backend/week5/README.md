## 왜 이 순서로 배우는가?

물류 서비스는 결국 아래 두 가지가 계속 일어나는 시스템이다.

- 상태가 바뀐다
- 같은 요청이 반복될 수 있다

예를 들어:

```text
출고 요청 생성
-> 재고 확인
-> 작업지시 생성
-> 작업 시작
-> 작업 완료
```

이 흐름에서 중요한 것은 단순 CRUD가 아니다.

- 지금 요청이 어느 단계에 있는지
- 같은 버튼을 두 번 눌렀을 때 시스템이 꼬이지 않는지
- 처음 만들 기능을 어디까지로 자를지

이 세 가지가 중요하다.

## 이번 주 학습 포인트

### 1. 상태 관리

상태는 "이 데이터가 지금 어느 단계에 있는가"를 표현한다.

예:

```text
requested -> allocated -> in_progress -> completed
```

상태를 관리해야 하는 이유:

- 잘못된 흐름을 막을 수 있다
- API 설계가 쉬워진다
- 테스트 포인트가 분명해진다
- 락이 필요한 지점이 보인다

### 2. 멱등성

멱등성은 같은 요청이 여러 번 와도 결과가 안전해야 한다는 뜻이다.

예:

- 출고 요청 생성 버튼을 두 번 눌렀다
- 네트워크가 느려서 사용자가 새로고침 후 다시 요청했다
- 서버 타임아웃 때문에 재시도 요청이 들어왔다

이때 작업지시가 두 개 생기면 안 된다.

### 3. MVP 설계

MVP는 가장 작은 문제 해결 세트다.

이번 주에는 "전체 물류 시스템"을 만들려 하지 않고,  
핵심 흐름 하나만 골라서 출발점을 정한다.
