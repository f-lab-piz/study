## MVP란?

MVP는 **가장 작은 문제 해결 세트**다.

중요한 점은 "기능이 적다"가 아니라  
**문제를 실제로 해결하는 최소 단위**라는 것이다.

---

## 왜 MVP가 필요한가?

프로젝트를 시작할 때 가장 흔한 문제는 이것이다.

> 만들고 싶은 것은 너무 많고, 어디서부터 시작할지 모르겠다.

MVP를 정하면 이 문제를 줄일 수 있다.

- 무엇을 먼저 만들지 정할 수 있다
- 무엇을 지금 만들지 말아야 하는지 정할 수 있다
- 백엔드와 프론트의 시작점이 보인다

---

## 이번 주 기본 흐름

이번 주에서는 아래 흐름을 기준으로 MVP를 자른다.

```text
출고 요청 생성 -> 재고 확인 -> 작업지시 생성 -> 상태 변경
```

이 흐름은 작지만 충분히 중요하다.

이 안에 이미 들어있는 것:

- 도메인
- 상태
- API
- 테스트 포인트
- 동시성 포인트

---

## MVP에 넣을 것과 뺄 것

### 넣을 것

- 상품 조회
- 재고 조회
- 출고 요청 생성
- 작업지시 생성
- 작업 상태 변경

### 뺄 것

- 로그인/권한관리
- 멀티 창고
- 입고 기능
- 알림
- 대시보드
- AI 자동 실행

처음에는 "좋아 보이는 기능"보다  
**핵심 흐름을 끝까지 확인할 수 있는 기능**이 중요하다.

---

## 서비스 한 줄부터 정하기

프로젝트를 시작할 때는 먼저 한 줄로 설명할 수 있어야 한다.

예:

```text
창고 담당자가 출고 요청을 등록하고 재고를 확인한 뒤 피킹 작업을 진행할 수 있는 시스템
```

이 문장이 있으면 아래가 쉬워진다.

- 어떤 사용자인지
- 어떤 데이터가 필요한지
- 어떤 화면이 필요한지
- 어떤 API가 필요한지

---

## 도메인 정하기

MVP에서는 엔티티를 많이 늘리지 않는다.

권장 시작:

- `Product`
- `Inventory`
- `OutboundRequest`
- `PickingTask`

이 정도면 핵심 흐름을 시작하기에 충분하다.

---

## 상태 정하기

MVP라고 해서 상태를 빼면 안 된다.

오히려 작은 서비스일수록 상태를 간단하게 정하는 것이 중요하다.

예:

### OutboundRequest

```text
requested -> allocated -> completed
              |
              -> failed
```

### PickingTask

```text
ready -> in_progress -> completed
```

---

## API 초안 정하기

MVP에서는 API도 작게 시작한다.

예:

```http
GET /products
GET /inventory
POST /outbound-requests
GET /outbound-requests/{id}
POST /picking-tasks
PATCH /picking-tasks/{id}/status
```

이 정도면:

- 데이터를 볼 수 있고
- 요청을 만들 수 있고
- 작업을 진행할 수 있다

---

## 첫 화면 2개만 고르기

처음부터 화면을 많이 만들 필요는 없다.

권장:

1. 출고 요청 등록 화면
2. 작업지시 목록/상태 변경 화면

이 두 개만 있어도 흐름을 끝까지 확인할 수 있다.

---

## 구현 순서

MVP는 기능 목록보다 **구현 순서**가 더 중요하다.

권장 순서:

1. `Product`, `Inventory`, `OutboundRequest`, `PickingTask` 정의
2. `POST /outbound-requests`
3. `GET /inventory`
4. `POST /picking-tasks`
5. `PATCH /picking-tasks/{id}/status`
6. 정상/실패 케이스 테스트

이 순서를 따르면:

- 데이터 구조가 먼저 잡히고
- 핵심 흐름이 살아나고
- 나중에 AI를 붙일 자리도 생긴다

---

## AI는 어디에 붙일까?

이번 단계에서 AI는 핵심 기능이 아니라 보조 기능으로 본다.

권장 위치:

- 재고 부족 사유 요약
- 출고 지연 사유 요약
- 운영자 질문에 대한 읽기 전용 답변

지금 단계에서 하지 않을 것:

- AI가 상태를 직접 바꾸는 기능
- AI가 재고를 직접 차감하는 기능
- AI가 작업 완료를 자동 승인하는 기능

---

## 정리

- MVP는 가장 작은 문제 해결 세트다
- 핵심 흐름 하나를 끝까지 연결할 수 있어야 한다
- 기능보다 먼저 사용자 흐름을 잡는 것이 좋다
- 처음에는 적게 만들고, 끝까지 도는 흐름을 만드는 것이 중요하다
